Framework 01

DORA — DevOps Research and Assessment

Founded 2014 · Google acquired 2018 · Annual State of DevOps Report

DORA quantifies software delivery performance. The core thesis from Accelerate (2018): software delivery performance directly predicts organisational performance — profitability, market share, and productivity.

The DORA Four Metrics

Deployment Frequency
How often code is deployed to production
Lead Time for Changes
Commit to production deployment time
Change Failure Rate
% of deployments causing production failures
MTTR
Time to restore service after a failure
Elite performers achieve both high throughput and high stability simultaneously. The old assumption that "speed trades off against quality" is empirically false.

Performance Bands

TierProfile
EliteDeploy on-demand; recover in < 1 hour; change failure rate < 5%
HighDeploy weekly–monthly; stable, predictable performance
MediumDeploying monthly; occasional outages
LowRare deployments; long recovery times; frequent failures

In 2023: ~18% Elite · ~23% High · ~44% Medium · ~15% Low

✓ Strengths

  • Empirically validated across 33,000+ professionals
  • Four metrics are simple, objective, and automatable
  • Ties engineering to business outcomes
  • Free survey instrument available at dora.dev

✗ Limitations

  • Measures delivery performance only
  • Can be gamed by low-value deployments
  • Does not address AI-augmented teams
  • MTTR is hard to measure consistently
Framework 02

SPACE — Developer Productivity Framework

Published 2021 · GitHub + Microsoft Research · ACM Queue

SPACE measures developer productivity across 5 dimensions. Core motivation: frustration with one-dimensional productivity measures (lines of code, commits/day) that capture activity but miss what actually matters.

The 5 Dimensions

DimensionWhat It CapturesExample Metrics
S — SatisfactionMeaning, burnout, tool satisfactionDeveloper satisfaction score, eNPS, burnout indicators
P — PerformanceOutcomes at individual/team/system levelPR acceptance rate, sprint goal achievement, reliability
A — ActivityVolume of engineering actionsCommits, PRs, code reviews, deployments
C — CommunicationCollaboration qualityPR review time, onboarding time, documentation quality
E — EfficiencyMinimal interruption and wasteBuild cycle time, meeting load, unplanned work %
Activity is the most dangerous dimension to over-index on. High activity does not mean high productivity.

3 Core Principles

PrincipleMeaning
Productivity is not one-dimensionalNo single metric captures it — measure across all dimensions
Measure at multiple levelsIndividual, team, and system levels often diverge
Perceptions matterSurvey data is not soft data — it is essential signal

Anti-patterns to Avoid

Anti-patternWhat It Looks LikeWhy Harmful
Proxy metric fixation"Commits per day is our KPI"Activity ≠ value; incentivises busy-work
Single-level measurementOnly measuring individual outputMisses systemic bottlenecks
Ignoring wellbeingNo burnout measurementLeads to retention crises
Activity as performanceMore PRs = better developersRewards quantity over quality
SPACE metrics should NEVER be used for individual performance reviews. This destroys psychological safety and causes gaming.

✓ Strengths

  • Prevents reductive, activity-based measurement
  • Includes developer wellbeing as first-class concern
  • Strong academic foundation (ACM Queue)
  • Applicable to AI-augmented teams

✗ Limitations

  • No standard benchmarks like DORA tiers
  • Survey dimensions require psychological safety
  • No prescribed metric set — inconsistency risk
  • Relatively new (2021); less tooling than DORA
Framework 03

CMMI — Capability Maturity Model Integration

Origin 1987 (CMM) · CMMI V2.0 2018 · V3.0 2023 · CMMI Institute

CMMI is a process improvement framework that defines what practices an organisation should have at each level of process maturity. Originally developed by the SEI at Carnegie Mellon for US DoD software contractors.

The 5 Maturity Levels

LevelNameCharacteristic
1InitialUnpredictable, poorly controlled, reactive
2ManagedProjects planned and tracked; reactive to problems
3DefinedOrganisation-wide standards; proactive management
4Quantitatively ManagedQuantitative quality/performance goals; predictable
5OptimisingContinuous improvement; defect prevention culture

Practice Areas (CMMI V2.0)

CategoryPractice Areas
DoingRequirements Development (RD), Technical Solution (TS), Product Integration (PI), Verification & Validation (VV)
ManagingProject Planning (PP), Project Monitoring & Control (PMC), Estimating (EST), Risk & Opportunity Management (RSK)
EnablingConfiguration Management (CM), Process Quality Assurance (PQA), Causal Analysis & Resolution (CAR)
ImprovingOrganisational Process Focus (OPF), Organisational Process Definition (OPD), Organisational Training (OT)
CMMI appraisals are a significant investment: $200K–$500K+ for a full Level 3 appraisal. Alternative: CMMI scoped appraisal or gap assessment ($10K–$50K).

CMMI + AI (2025 Update)

The CMMI AI Working Group (IBM + CMMI Institute, announced April 2025) is integrating GenAI into CMMI practice areas — directly relevant to the PhD research gap on AI-augmented SPI.

✓ Strengths

  • Most comprehensive process framework available
  • Internationally recognised certification
  • Proven ROI in large-scale regulated environments
  • Being actively extended for AI (CMMI AI WG)

✗ Limitations

  • Very high implementation cost ($200K–$500K+)
  • 12–36 months to first appraisal result
  • Can become bureaucratic box-ticking
  • Assumes human-authored artefacts; no AI-actor model yet
Framework 04

SonarQube — Code Quality & Security Platform

SonarSource · Community (free) + Developer/Enterprise editions · sonarqube.org

SonarQube is a static analysis platform that continuously inspects code quality and security. It integrates into CI/CD pipelines to act as an automated quality gate before code reaches production.

The 5 Quality Dimensions

Bugs
Definite errors causing incorrect behavior
Vulnerabilities
Security weaknesses exploitable by attackers
Code Smells
Maintainability issues increasing future change cost
Coverage
% of code exercised by automated tests
Duplication
% of code that is duplicated blocks

Quality Gates

A Quality Gate is a pass/fail policy applied to every analysis. The default "Sonar Way" gate fails if new code has: any bugs, any vulnerabilities, any security hotspots not reviewed, coverage < 80%, or duplication > 3%.

Reliability & Security Ratings

RatingReliability (Bugs)Security (Vulnerabilities)
ANo bugsNo vulnerabilities
BAt least 1 minor bugAt least 1 minor vulnerability
CAt least 1 major bugAt least 1 major vulnerability
DAt least 1 critical bugAt least 1 critical vulnerability
EAt least 1 blocker bugAt least 1 blocker vulnerability

✓ Strengths

  • 1–3 days to first value — fastest of all four frameworks
  • CI/CD integration acts as automated quality gate
  • Supports 30+ languages
  • Free Community Edition covers most use cases

✗ Limitations

  • Static analysis only — misses runtime behavior
  • No measurement of cognitive or intent debt
  • Coverage metrics can be gamed by shallow tests
  • Misses semantic/intent issues in AI-generated code

Decision Tree

Need government/defence certification? YES → CMMI Level 3+ required NO → Continue ↓ Primary concern: how fast and reliably you ship? YES → Start with DORA NO → Continue ↓ Primary concern: developer wellbeing / retention? YES → Use SPACE (pair with DORA) NO → Continue ↓ Primary concern: code quality, security, or debt? YES → Deploy SonarQube immediately NO → Continue ↓ Building a comprehensive improvement program? YES → All four in sequence: 1. SonarQube (immediate code quality baseline) 2.
