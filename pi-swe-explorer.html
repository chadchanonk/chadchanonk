What is Process Improvement in SE?

Software Process Improvement (SPI) is the discipline of systematically analysing, measuring, and enhancing the practices used to build software — from requirements gathering through deployment and maintenance. Research in this area bridges empirical software engineering, organisational management, and quality science.

🎯

Reduce defects, cycle time, and cost while increasing predictability, quality, and developer experience across the SDLC.

Key Research Findings

Technical Debt Paradox (2025)
AI systems may reduce traditional technical debt through automated refactoring while simultaneously accelerating "cognitive debt" and "intent debt" — where code works but teams no longer understand why it was built that way. (arXiv 2603.22106)
AI Code Quality Risk
GitClear's 2025 report documents an eightfold increase in duplicated code blocks since widespread AI coding adoption — a clear maintainability warning signal for SPI researchers.
Financial Sector Case Study
A global financial services firm deploying AI-powered code analysis achieved a 25% improvement in developer efficiency and significant reduction in system outages through proactive technical debt management.
Experienced Developer Productivity
Research indicates AI-assisted programming can decrease experienced developer productivity by increasing technical debt and maintenance burden, despite perceived short-term speed gains. (arXiv 2510.10165)

Historical Timeline of SPI

Key milestones in the evolution of Software Process Improvement — from foundational quality science to the AI-augmented era.

1950s–60s
Quality Science Foundations
Deming, Juran, and Crosby establish Statistical Process Control, PDCA cycle, and quality cost models — later adopted by software engineering.
1962
Toyota Production System (TPS)
Toyota formalises Lean manufacturing, Kanban, and the Kaizen philosophy — foundations of modern software flow management.
1986
Six Sigma — Motorola
Motorola formalises Six Sigma. GE, then software organisations, adopt DMAIC as a structured defect-reduction discipline.
1991
CMM (Capability Maturity Model)
SEI releases the CMM for software. The 5-level maturity model becomes the dominant SPI framework for large software organisations globally.
1995
ISO/IEC 15504 (SPICE)
International standard for software process assessment, enabling cross-border SPI comparability and audit frameworks.
2001
Agile Manifesto
12 principles shift the SPI conversation from heavyweight process prescriptions to iterative, people-centric improvement through Scrum, XP, and Crystal.
2002
CMMI v1 (Integration)
SEI merges multiple CMMs into CMMI, covering software, systems, and supplier management. Global adoption accelerates especially in government contracting.
2009
DevOps Movement
Patrick Debois coins "DevOps" — combining development and operations through CI/CD pipelines, automated testing, and shared responsibility culture.
2014
DORA Research Program
DevOps Research and Assessment begins quantifying elite software delivery performance through deployment frequency, lead time, MTTR, and change failure rate.
2018
SAFe 5 / Scaled Agile
Scaled Agile Framework reaches mainstream enterprise adoption, bridging Agile team practices with portfolio-level SPI governance.
2022–24
GenAI Enters the SDLC
GitHub Copilot, ChatGPT, and code-generation LLMs transform developer workflows — triggering new SPI research on AI code quality, cognitive debt, and measurement.
2025
CMMI AI Working Group
IBM joins CMMI Institute's AI WG, embedding RPA, ML, GenAI, and DevSecOps into next-generation CMMI process areas for AI-augmented development.

Open Research Questions

Open Problem · Measurement
How should process quality metrics evolve when significant portions of code are AI-generated? Existing DORA and SPACE frameworks were not designed for AI-augmented teams.
Open Problem · Ethics & Skills
Organizational expectations for rapid AI productivity gains, without corresponding investment in learning support, may erode the deep understanding needed for safe, long-term code evolution.
Open Problem · Data Quality
AI-for-SPI approaches depend on high-quality historical process data — a persistent challenge for organisations with immature process instrumentation and telemetry.
